# システム概要

## 目的

Electric Sheep は、若いうちから日々の出来事、思い出、感情、写真、音声を保存し、何十年後の自分や家族が忘れていた記憶にたどり着けるようにするライフログサービスである。

主な目的は以下とする。

- 日々の思い出を文章・写真・音声・感情・タグとともに保存する。
- 保存した記憶を後から時系列、タグ、感情、自然文検索で見返せるようにする。
- 親族や親しい人との会話のきっかけを作る。
- 投稿継続を促す仕組みにより、長期利用しやすいサービスにする。
- ユーザー自身の投稿データとメディアファイルを標準的な形式で取り出せるようにする。

## 領域（スコープ）

### MVPで含める機能

- ユーザー登録、ログイン、ログアウト
- 思い出投稿の作成
- 投稿本文、感情、タグ、投稿日時、公開範囲の保存
- 写真・音声などメディアファイルの保存
- 自分の投稿一覧、投稿詳細の閲覧
- 投稿数、連続投稿日数などの簡易統計
- ヘルスチェック API

### 段階的に追加する機能

- 生成AIを用いた自然文検索、要約
- 感情ログの集計、感情グラフ
- 「一年前の今日」「ランダムな思い出」などの通知
- 相互フォロワーへの投稿共有
- 投稿データとメディアファイルを含むエクスポート

### 除外する範囲

- 投稿の編集・削除機能は初期MVPでは必須範囲外とする。
- 一般公開SNSのような不特定多数への拡散機能は対象外とする。
- 医療・心理診断のような専門的判断は提供しない。
- 生成AIの回答を記憶の事実として確定する機能は提供しない。

## 利用者（ユーザ）

| 利用者 | 役割 |
| --- | --- |
| 一般ユーザー | 自分の日々の思い出を投稿し、後から閲覧する。 |
| 相互フォロワー | 許可された投稿のみ閲覧する。 |
| 家族・親しい人 | 将来的に本人が共有した記憶を一緒に見返す。 |
| 管理者 | サービス運用、障害対応、不正利用対応を行う。 |

## 主要機能

| 機能 | 概要 | 優先度 |
| --- | --- | --- |
| 認証 | ユーザー登録、ログイン、ログアウト、アクセストークン発行 | 高 |
| 投稿 | 思い出本文、感情、タグ、メディア、公開範囲を保存 | 高 |
| 閲覧 | 自分の投稿一覧、投稿詳細を取得 | 高 |
| メディア保存 | 写真、動画、音声ファイルを保存し、投稿に紐づける | 高 |
| 継続支援 | 連続投稿日数、総投稿数を表示 | 中 |
| 共有 | 相互フォロワーに限定して投稿を共有 | 中 |
| 検索・要約 | 生成AIを用いて自然文検索・要約を行う | 中 |
| 通知 | 過去の投稿閲覧を促す通知を送る | 低 |
| エクスポート | ユーザーの投稿データとメディアファイルをZIP形式で出力 | 低 |

## 非機能要件

| 分類 | 要件 |
| --- | --- |
| パフォーマンス | 通常APIは2秒以内の応答を目標とする。 |
| AI処理 | 生成AI検索・要約は通常APIと分け、30秒以内の応答を目標とする。 |
| 可用性 | 投稿データ、メディア、DBはバックアップを取得し、復旧可能な状態を保つ。 |
| セキュリティ | 個人的な記憶や写真を扱うため、認証、認可、暗号化、公開範囲制御を重視する。 |
| 保守性 | ドメイン単位で処理を分離し、理解しやすい構成にする。 |
| 移行性 | 将来別サービスへ移行できるよう、データ構造を明確にし、JSONとメディアファイルをまとめたエクスポートに対応する。 |
| UI連携 | 100年使うことを想定したシンプルな画面構成を前提に、APIも複雑にしすぎない。 |

## 前提条件

- バックエンドは Go 1.22 + Gin を利用する。
- API は REST 形式とし、JSON を入出力の基本形式とする。
- DB は Neon PostgreSQL を利用する。
- DB設計は標準的な PostgreSQL 機能を中心に行い、将来的に RDS や Cloud SQL などの PostgreSQL 互換DBへ移行できるようにする。
- メディアファイルの本番保存先は Cloudflare R2 とする。R2 は S3互換APIを持つオブジェクトストレージとして利用する。
- 開発環境ではローカルストレージまたは MinIO を利用できるようにし、本番環境との差し替えは設定で行う。
- メディア移行性を確保するため、DBには一時的な公開URLではなく、保存先キー、元ファイル名、MIMEタイプ、ファイルサイズ、チェックサムを保存する。
- データエクスポートでは投稿JSONとメディアファイル本体を同じZIPに含め、別サービスへ移行しても投稿とメディアの紐づけを復元できるようにする。
- 認証は JWT Bearer Token 方式を採用する。
- ログイン時に短期有効の `access_token` と長期有効の `refresh_token` を発行し、`refresh_token` はハッシュ化してDBに保存する。
- 生成AI連携は外部AI APIを利用する前提とし、APIキーなどの秘匿情報は環境変数で管理する。
